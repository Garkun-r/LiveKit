# Парсер тегов и навыки робота

Документ описывает служебные теги, которые LLM может добавить в ответ
робота, и действия, которые агент выполняет после озвучивания очищенного
ответа.

## Цель

LLM отвечает в формате:

```text
Текст для клиента. [STATUS: END]
```

Клиент слышит и видит только обычный текст:

```text
Текст для клиента.
```

Служебный тег не попадает в TTS и клиентскую текстовую транскрипцию. Raw-ответ
с тегом сохраняется только в служебном событии `tag_events`, чтобы можно было
разобрать, почему сработал навык.

## Интеграция с LiveKit

Реализация использует pipeline nodes LiveKit:

- `Assistant.llm_node` накапливает raw LLM output, парсит теги и привязывает
  результат к текущему `SpeechHandle`.
- `Assistant.tts_node` применяет streaming-sanitizer, чтобы квадратные сегменты
  `[...]` не ушли в синтез речи.
- `Assistant.transcription_node` применяет тот же sanitizer, чтобы теги не
  ушли в видимый текст и assistant history/transcript.
- Навык запускается через callback завершения `SpeechHandle`.

Если клиент перебил робота, действие все равно выполняется, если тег уже дошел
до парсера. Если генерация была отменена до появления тега, выполнять нечего.

Основные файлы:

- `agents/main-bot/src/robot_tags.py` - parser и sanitizer.
- `agents/main-bot/src/robot_skills.py` - runtime навыков.
- `agents/main-bot/src/agent.py` - подключение parser/sanitizer к LiveKit
  pipeline и экспорт `tag_events`.

## Поддержанные теги

| Тег | Что слышит клиент | Дальнейшее действие |
| --- | --- | --- |
| `[STATUS: END]` | Очищенный текст без тега | После завершения ответа запросить завершение звонка внутренней логикой агента. `end_call` tool не используется. |
| `[STATUS: SPAM]` | Очищенный текст без тега | То же, что `END`, но причина в событии будет `status_spam`. |
| `[STATUS: SMS_LINK]` | Очищенный текст без тега | Placeholder-навык. Сейчас только записывает событие с телефоном клиента и пустым `message_template`. SMS не отправляется. |
| `[TRANSFER: ID]` | Очищенный текст без тега | Placeholder-навык. Сейчас записывает `ID`, room, participant identity и телефон клиента. Перевод через Asterisk/LiveKit не выполняется. |
| `[GEO_SEARCH: город компании, объект/улица]` | Очищенный текст без тега | Placeholder-навык. Сейчас записывает исходный запрос, город и объект/улицу. Внешний GEO API не вызывается. |

Старые или неподдержанные теги, например `[STATUS: LEAD]`, скрываются из TTS и
транскрипции, но действие не запускают.

## Правила парсинга

- Все квадратные сегменты `[...]` скрываются из TTS и клиентской транскрипции.
- Первый валидный action-tag выигрывает.
- Остальные валидные action-tags скрываются и записываются в `ignored` с
  причиной `additional_action_tag`.
- Невалидные или неизвестные квадратные сегменты скрываются и записываются в
  `ignored` с причиной `unsupported_or_non_action_tag`.
- Если очищенный текст перед первым валидным action-tag заканчивается на `?`,
  действие не выполняется. Такой тег записывается в `ignored` с причиной
  `question_before_tag`, а клиенту отправляется только вопрос.
- Парсер не выполняет внешние действия. Он только возвращает структурированный
  результат. Все side effects живут в `robot_skills.py`.

Пример с вопросом:

```text
Вам удобно получить ссылку по SMS? [STATUS: SMS_LINK]
```

Клиент получает:

```text
Вам удобно получить ссылку по SMS?
```

Навык SMS не запускается, потому что перед тегом стоит вопрос.

## Экспорт tag_events

В n8n export добавляется `tag_events`. Каждое событие содержит:

- `raw_text` - полный ответ LLM с тегами.
- `clean_text` - текст после удаления квадратных сегментов.
- `selected` - выбранный тег или `null`.
- `ignored` - скрытые, но не исполненные теги с причиной игнорирования.
- `interrupted` - был ли перебит соответствующий `SpeechHandle`.
- `context` - агент, room, participant identity и SIP-номера.
- `skill_result` - результат выполнения навыка или причина `ignored`.

Эти данные служебные. Их можно использовать для диагностики и последующей
интеграции SMS, Asterisk transfer и GEO API, но они не должны озвучиваться
клиенту.

## Целевая логика навыков

Этот раздел фиксирует контракт для следующего этапа реализации навыков. Сейчас
runtime в `robot_skills.py` оставляет SMS, transfer и GEO как placeholder, чтобы
не менять production-телефонию без отдельной проверки.

### Общий lookup клиента

Для навыков, которым нужны данные компании, входной ключ - DID проекта из SIP
атрибутов LiveKit:

- `jcall.did`;
- `x-did`;
- `X-DID`;
- `sip.h.X-DID`;
- `sip.trunkPhoneNumber`.

На VPS DID хранится в `voicebot.public."CallerID"`. Связь с клиентом:

```sql
select ci.client_id
from "CallerID" ci
where regexp_replace(coalesce(ci."CallerID", ''), '\D', '', 'g') = $1
limit 1;
```

Дальше `client_id` используется для `clients`, `transfer_number` и `locations`.

### TRANSFER

LLM может прислать:

- прямой номер: `[TRANSFER: +79990000000]`, `[TRANSFER: 89990000000]`;
- код направления: `[TRANSFER: TR1]`.

Правило разбора:

1. Если payload после удаления нецифровых символов похож на телефонный номер,
   skill считает это прямым номером.
2. Если payload не похож на телефонный номер, skill считает его кодом/описанием
   направления и ищет номер в `transfer_number`.

Поиск направления:

```sql
select transfer_code
from transfer_number
where client_id = $1
  and disc = $2
limit 1;
```

`disc` - значение из тега, например `TR1`. `transfer_code` в текущей базе
хранится как `+7...`; перед передачей в Asterisk номер нужно нормализовать до
11 цифр без `+`.

Критичный Asterisk-контракт старого робота:

- `EXTEN` - номер назначения;
- SIP `From` - номер звонящего клиента;
- SIP `Diversion` - DID проекта;
- дальше Asterisk `from-robot` выставляет `CALLERID(rdnis)`,
  `REDIRECTING(...)`, пишет запись и делает исходящий `Dial` через Beeline.

Чтобы сохранить эту схему, LiveKit transfer не должен идти напрямую в Beeline.
Нужно вернуть вызов на текущий Asterisk через локальный LiveKit SIP/Asterisk
контур и собрать совместимый вход в отдельном Asterisk context `from-livekit`.

Предлагаемая схема:

```text
LiveKit skill TRANSFER
  -> LiveKit SIP transfer/create SIP leg to local Asterisk
  -> Asterisk context from-livekit
  -> нормализация DID, caller phone, transfer target
  -> переход в старую схему from-robot
  -> Dial(PJSIP/8${EXTEN:1}@beeline,60)
```

`from-livekit` должен привести вход от LiveKit к тому же состоянию, которое
сейчас ожидает `from-robot`. Если LiveKit не сможет поставить `Diversion`,
`from-livekit` должен взять DID из `X-DID`/служебного header и затем передать
его в старую логику так, чтобы итоговая схема отправки в Beeline осталась
идентичной `from-robot`.

Тестовый DID для будущей проверки transfer - Beeline-номер, который оканчивается
на `88` и сейчас отправляет звонок на LiveKit. Сам transfer пока не тестируем.

### SMS_LINK

Skill отправляет SMS на телефон звонящего клиента из `sip.phoneNumber`.
Сообщение повторяет текущую n8n-логику:

```text
<company_name>
Наш сайт: <company_website без http(s):// и www.>
<SMS_LINK_SIGNATURE>
```

Данные компании:

```sql
select company_name, company_website
from clients
where id = $1
limit 1;
```

Конфигурация хранится только в env или LiveKit Cloud secrets:

```env
SMS_RU_API_URL=https://sms.ru/sms/send
SMS_RU_API_ID=
SMS_LINK_SIGNATURE=Отправил наш ИИ робот: jcall.io
```

Реальный `SMS_RU_API_ID` нельзя хранить в git. Если ключ был отправлен в чат
или n8n export, его нужно считать раскрытым и заменить перед production.

После успешной постановки SMS skill завершает звонок как `END`.

### GEO_SEARCH

Skill получает payload вида:

```text
[GEO_SEARCH: город компании, объект/улица]
```

Целевая логика:

1. Получить координаты названного клиентом места через Google Places API.
2. Выбрать все филиалы клиента из `locations`:

   ```sql
   select description, adress, latitude, longitude
   from locations
   where client_id = $1;
   ```

3. Посчитать расстояние Haversine до каждого филиала.
4. Взять только ближайший филиал.
5. Если расстояние больше `50` км, ответить шаблоном:

   ```text
   Я не нашла рядом с вами наш филиал, попробуйте назвать адрес иначе.
   ```

6. Если филиал рядом, ответить шаблоном без дополнительного LLM-запроса:

   ```text
   Ближайший филиал находится по адресу <adress>, примерно <distance> от вас.
   Хотите, соединю вас с этим адресом?
   ```

Для расстояния меньше одного километра лучше говорить метры, иначе километры.
GEO skill продолжает диалог и сам по себе не переводит звонок. Перевод
происходит только если следующий ответ LLM выберет `TRANSFER`.

## Добавление нового тега

1. Добавить распознавание формата в `robot_tags.py`.
2. Добавить действие в `RobotSkillRunner._run_selected`.
3. Добавить unit tests для parser/sanitizer и runtime навыка.
4. Проверить, что новый тег не попадает в TTS и транскрипцию.
5. Если навык делает внешний вызов, держать интеграцию в отдельном модуле или
   в подходящем shared helper, а parser оставить чистым.
